# 离线工具注意事项 v1

## 1. 最核心注意事项

这个工具的本质是“离线本地仓库生成工具”，不是简单的 rpm/deb 下载脚本。开发时必须始终围绕下面这个目标：

> 下载端生成的离线包复制到目标离线服务器后，即使目标服务器完全不能联网，也应能通过本地仓库完成安装。

因此，任何只下载显式包、不解析完整依赖、不做离线 dry-run 的方案，都存在上线风险。

## 2. 包组下载注意事项

包组下载是最大风险点。

选择包组时，必须下载：

- 包组显式包含的包；
- 包组 mandatory 包；
- 包组 default 包；
- 用户策略要求的 optional 包；
- 所有 Requires / Depends / Pre-Depends；
- 需要时包含 weak dependencies / Recommends；
- module stream 相关包；
- post-install 阶段可能需要的基础工具。

不能只根据 `group info` 或配置文件里的包名列表打包。

## 3. 下载机已安装依赖导致漏包

这是最容易忽略的问题。

如果下载机已经安装了某些依赖，普通 download --resolve 可能不会把这些依赖下载下来。离线服务器上没有这些依赖时，安装就会失败。

建议：

- RPM 系使用空白 installroot；
- DEB 系使用空白 dpkg status；
- 打包前使用本地仓库 dry-run 验证。

## 4. 目标环境选择注意事项

下载模式必须让用户选择目标环境，而不是默认使用下载机环境。

必须至少选择：

- OS 类型；
- OS 版本；
- SP 或 minor 版本；
- 目标架构；
- 包管理类型；
- 仓库配置。

尤其要支持在 x86 下载机上下载 aarch64 离线包。

## 5. 架构兼容注意事项

架构必须严格校验。

常见映射：

| 系统显示 | 包仓库架构 |
|---|---|
| x86_64 | x86_64 / amd64 |
| aarch64 | aarch64 / arm64 |
| loongarch64 | loongarch64 |
| riscv64 | riscv64 |

RPM 和 DEB 的架构名称可能不一致，不能直接混用。

## 6. 仓库源注意事项

镜像源不能只测 ping 或延迟。

必须检查：

- 目标版本目录是否存在；
- 目标架构目录是否存在；
- repodata 或 Packages.gz 是否存在；
- GPG key 是否可用；
- 包是否真的存在；
- 源是否过期或不完整。

延迟最低的源不一定是可用源。

## 7. `/tmp` 和工作目录注意事项

不建议默认自动扩容 `/tmp`。

更稳妥的做法：

- 使用工具目录下的 workdir；
- 支持在 global.conf 中配置 WORKDIR；
- 下载前检查空间；
- 空间不足时提示换目录或清理空间；
- 只有在明确检测到 LVM 且用户确认后，才允许扩容。

## 8. 安装方式注意事项

不建议直接执行：

```bash
rpm -Uvh *.rpm
```

或：

```bash
dpkg -i *.deb
```

推荐通过本地仓库安装。

RPM：

```bash
dnf --disablerepo="*" --enablerepo="local-offline" install <package>
```

DEB：

```bash
apt-get update
apt-get install <package>
```

这样依赖顺序和版本选择由包管理器处理。

## 9. manifest 注意事项

离线包必须包含 manifest.json。安装端不能靠目录名判断离线包是否可用。

manifest 至少应包含：

- 工具版本；
- 包类型；
- 目标 OS；
- 目标版本；
- 目标 SP；
- 目标架构；
- 仓库来源；
- 依赖策略；
- 选中的包或包组；
- 解析后的完整包清单；
- checksum；
- 创建时间。

## 10. 增量打包注意事项

增量打包容易污染已有离线包。

只有以下信息全部一致时才允许增量：

- OS；
- VERSION；
- SP；
- ARCH；
- PKG_TYPE；
- REPO_ID；
- module stream；
- 依赖策略；
- GPG 策略。

否则必须重新打包。

增量后必须重新生成仓库元数据，并重新做 dry-run 验证。

## 11. 版本比较注意事项

不能用字符串比较版本。

错误示例：

```text
1.10 被误判为小于 1.9
```

RPM 应使用 RPM 的 EVR 规则或 `rpmdev-vercmp`。

DEB 应使用：

```bash
dpkg --compare-versions
```

## 12. 安全校验注意事项

生产环境离线安装通常以 root 权限执行，必须重视安全校验。

建议至少实现：

- SHA256 校验；
- GPG 校验；
- repo metadata 校验；
- manifest 校验；
- 包数量校验；
- 安装前 dry-run；
- 安装前二次确认；
- 完整日志留存。

## 13. 失败处理注意事项

失败时不要直接清空现场。

应保留：

- workdir；
- 下载缓存；
- 日志；
- manifest；
- resolved package list；
- 失败命令；
- 返回码；
- 错误输出。

这样方便继续增量下载或定位问题。

## 14. 建议开发顺序

第一阶段必须先完成：

1. 目标环境选择；
2. 仓库配置加载；
3. 单包完整依赖下载；
4. 包组完整依赖下载；
5. 本地仓库生成；
6. dry-run 验证；
7. manifest 和 checksum；
8. 安装端读取 manifest 并通过本地仓库安装。

第二阶段再做：

1. 增量打包；
2. 多镜像自动切换；
3. GPG 完整校验；
4. 安装报告；
5. 回滚提示；
6. 图形化或菜单增强。

## 15. 上线前测试清单

上线前至少测试以下场景：

- x86 下载 x86 包；
- x86 下载 aarch64 包；
- 单包下载；
- 包组下载；
- 下载机已安装依赖但离线机未安装依赖；
- 目标源缺包；
- 源 404；
- 空间不足；
- 离线包复制后 checksum 不一致；
- OS 版本不匹配；
- 架构不匹配；
- 增量打包混入不同版本包；
- 本地仓库 dry-run 失败；
- 安装中断后重新执行。
