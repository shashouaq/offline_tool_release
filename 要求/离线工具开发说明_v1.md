# 离线工具开发说明 v1

## 1. 工具目标

本工具用于在一台联网的相近环境服务器或虚拟机中，按目标离线服务器的 OS、版本、SP、架构和包类型，下载指定包或包组及其完整依赖，生成可迁移的离线安装包。离线包复制到目标服务器后，应能通过本地仓库完成安装、校验和结果输出。

核心目标不是“下载几个 rpm/deb 文件”，而是生成一个可以在目标离线环境中稳定安装的本地仓库交付包。

## 2. 核心业务边界

必须明确区分两个环境：

| 环境 | 说明 | 是否联网 |
|---|---|---|
| 下载机环境 | 运行下载和打包逻辑的机器，可以是 x86 服务器或虚拟机 | 是 |
| 目标离线环境 | 最终安装离线包的服务器 | 否 |

下载机环境只能作为运行平台，不应直接决定下载包的 OS、版本和架构。下载模式必须由用户选择目标环境，例如：openEuler 22.03 SP4 aarch64、RHEL 7.9 x86_64、Ubuntu 22.04 amd64。


## 5. 依赖解析原则

### 5.1 不允许只下载显式包

选择包组时，不能只下载包组中显式列出的包。必须解析完整依赖树，包括：

- Depends；
- Pre-Depends；
- Requires；
- weak dependencies，按策略决定是否纳入；
- recommends，按策略决定是否纳入；
- module stream；
- group mandatory/default/optional 包；
- post-install 阶段需要的基础工具。

### 5.2 必须避免下载机已安装依赖导致漏包

常见错误是直接在下载机上执行：

```bash
dnf download --resolve <package>
```

或：

```bash
yumdownloader --resolve <package>
```

如果下载机已经安装某些依赖，这些依赖可能不会被下载，复制到离线服务器后会缺包。

推荐方式：

- RPM 系：使用空白 installroot 解析依赖；
- DEB 系：使用空白 dpkg status 或专用离线解析目录解析依赖。

## 6. RPM 系推荐实现策略

### 6.1 单包下载

推荐逻辑：

```text
指定目标 repo
↓
指定目标 arch
↓
创建空白 installroot
↓
使用 dnf/yum 在 installroot 中做 install/downloadonly
↓
收集下载结果
↓
生成 repodata
```

示例方向：

```bash
dnf \
  --installroot "$WORKDIR/installroot" \
  --releasever "$TARGET_RELEASEVER" \
  --setopt=install_weak_deps=True \
  --downloadonly \
  --downloaddir "$CACHE_DIR/packages" \
  install -y <package>
```

### 6.2 包组下载

RPM 包组要区分两类：

| 类型 | 说明 |
|---|---|
| 自定义包组 | 工具维护 groups.conf，包组映射到具体包列表 |
| 仓库原生包组 | 使用 dnf/yum group 机制解析，例如 Development Tools |

原生包组要注意 comps.xml、group id、mandatory/default/optional、module stream。

### 6.3 本地仓库生成

RPM 离线包必须包含：

```text
packages/
repodata/
manifest.json
checksum.sha256
install.sh
verify.sh
README.md
```

生成元数据：

```bash
createrepo_c packages/
```

安装时应通过本地 repo 安装，不建议直接 `rpm -Uvh *.rpm`。

## 7. DEB 系推荐实现策略

DEB 系没有完全等价于 RPM group 的机制，包组建议使用：

- 自定义包组；
- meta package；
- tasksel task；
- apt pattern。

依赖解析需覆盖：

- Depends；
- Pre-Depends；
- Recommends，按策略决定；
- meta package 引出的依赖。

本地仓库建议结构：

```text
pool/
dists/
manifest.json
checksum.sha256
install.sh
verify.sh
README.md
```

安装端通过本地 apt 源安装，不建议直接 `dpkg -i *.deb`。

## 8. manifest.json 设计建议

离线包必须包含 manifest.json，安装端应以它作为唯一可信入口。

示例：

```json
{
  "tool_version": "v6.x",
  "package_type": "rpm",
  "target_os_id": "openEuler",
  "target_version_id": "22.03",
  "target_sp": "SP4",
  "target_arch": "x86_64",
  "created_at": "2026-04-25T00:00:00+08:00",
  "dependency_policy": {
    "include_weak_deps": true,
    "include_recommends": true
  },
  "repo_sources": [],
  "selected_items": [],
  "resolved_packages": [],
  "checksums": {},
  "verify_commands": []
}
```

## 10. 兼容性判断策略

| 级别 | 条件 | 动作 |
|---|---|---|
| 完全匹配 | OS、版本、SP、架构、包类型全部一致 | 默认允许 |
| 兼容匹配 | 同 OS 大版本、同架构，但 SP/minor 不完全一致 | 强警告后允许 |
| 不兼容 | OS、架构或包类型不同 | 禁止安装 |

安装端不能只靠目录名判断兼容性，必须读取 manifest.json。

## 11. 增量打包策略

增量打包前必须比较已有离线包 manifest：

- package_type 是否一致；
- target_os_id 是否一致；
- target_version_id 是否一致；
- target_sp 是否一致；
- target_arch 是否一致；
- repo_sources 是否一致；
- dependency_policy 是否一致；
- module stream 是否一致。

不一致时禁止增量，只能重新打包。

增量完成后必须重新生成仓库元数据，并重新执行本地仓库 dry-run 验证。

## 12. 日志和审计

建议日志内容包括：

- 时间戳；
- 执行命令；
- 返回码；
- 标准输出和错误输出摘要；
- 下载包数量；
- 成功包列表；
- 失败包列表；
- 失败原因；
- 仓库地址；
- 目标 OS / 架构；
- 最终离线包路径；
- 校验结果。

建议输出：

```text
logs/YYYYMMDD.log
output/<offline_package>/manifest.json
output/<offline_package>/resolved_packages.lock
output/<offline_package>/checksum.sha256
output/<offline_package>/install_report.txt
```

## 13. 开发优先级

| 优先级 | 内容 |
|---|---|
| P0 | 区分下载机环境和目标环境 |
| P0 | 包/包组完整依赖解析 |
| P0 | 空白 installroot / 空白 dpkg status 依赖闭环 |
| P0 | 本地仓库 dry-run 验证 |
| P0 | manifest.json 兼容性判断 |
| P1 | 本地 repo 安装，避免直接 rpm/dpkg |
| P1 | 增量打包一致性校验 |
| P1 | checksum 和日志审计 |
| P2 | GPG 校验、签名、离线包加固 |
| P2 | 失败回滚提示和环境恢复 |

## 14. 开发验收标准

至少满足以下条件才算达到第一版可用：

1. 能在联网 x86 机器上选择目标 OS、版本、SP 和架构。
2. 单包下载能包含完整依赖。
3. 包组下载能包含完整依赖。
4. 离线包内包含可用本地仓库元数据。
5. 打包前能用本地仓库执行 dry-run 验证。
6. 离线端能读取 manifest 并判断兼容性。
7. 离线端安装时只启用本地 repo。
8. 失败时能输出明确原因和日志路径。
