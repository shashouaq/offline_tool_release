# 离线工具平台 v14.0 使用说明书

## 1. 文档目标

本文档用于指导运维、测试和交付人员使用 `offline_tools_v14.sh` 完成以下工作：

1. 在联网环境中按目标操作系统和架构下载工具包或包组。
2. 生成可在离线环境直接使用的本地仓库离线包。
3. 在完全离线的目标环境中基于本地仓库完成安装。
4. 在出现问题时通过日志、校验文件和元数据快速定位原因。

本文档基于当前项目目录：

`D:\arm\offline_tool\offline_tool_release`

主脚本入口：

`offline_tools_v14.sh`

---

## 2. 产品定位

该工具不是“简单下载 RPM/DEB 文件”的脚本，而是一个完整的离线仓库交付工具。它的目标是：

- 在联网机器上，针对用户选择的目标 OS/版本/架构下载工具及依赖。
- 自动构建本地仓库索引。
- 打包为一个可移植的离线包。
- 在离线机器上只使用离线包内部仓库完成安装。

当前版本已经引入以下关键机制：

- `manifest.json` 作为离线包的安装兼容性可信来源。
- RPM/DEB 本地仓库索引构建。
- 双语菜单基础框架。
- 自主远程回归能力。
- 下载、打包、安装的结构化日志。

---

## 3. 适用场景

适合以下典型场景：

- 生产环境服务器不能访问外网，需要预下载工具并带入。
- 一台联网构建机要为多种目标系统生成不同离线包。
- 交付人员需要把同一工具集提供给多台同版本服务器。
- 需要长期维护离线包版本并重复安装、升级、回归验证。

不建议直接用于以下场景：

- 当前机器和目标机器系统不一致，但未明确选择目标 OS/架构。
- 将不同 OS 或不同架构的工具混装到同一个离线包。
- 仅希望下载个别单包，而不关心依赖闭包和离线可安装性。

## 5. 运行前准备

### 5.1 联网下载环境准备

要求：

- 具备稳定网络访问目标仓库能力。
- 具备 `dnf/yum` 或 `apt-get`。
- 具备 `tar`、`curl`。
- RPM 环境建议具备 `createrepo_c`。
- DEB 环境建议具备 `dpkg-dev`。

建议：

- 预留足够 `/tmp` 或工作目录空间。
- 使用 root 或具备足够包管理权限的账号。
- 确认目标 OS、版本、架构，而不是默认使用当前系统。

### 5.2 离线安装环境准备

要求：

- 不依赖外部网络。
- 能够使用本地 `dnf/yum` 或 `apt-get`。
- 将离线包拷贝到目标机器可访问目录。
- 解包空间足够。

### 5.3 `/tmp` 空间说明

主脚本内置了 `/tmp` 空间检查逻辑：

- 当 `/tmp` 空间不足时，会尝试切换工作目录。
- root 环境下，如果满足条件，会尝试临时扩展 tmpfs 或切换到新的 tmpfs 工作区。
- 非 root 环境下，会切换到回退目录，例如 `/var/tmp/offline_tools_v14`。

建议在大包组下载前预留 20GB 以上临时空间。

## 7. 下载模式使用流程

### 7.1 目标选择

下载模式首先要求选择目标环境，而不是默认当前系统。目标选择包括：

- 目标操作系统
- 目标架构
- 是否跳过 SSL 证书校验

当前业务规则要求：

- 不同 OS/版本/架构不能混包。
- `x86_64` 与 `aarch64` 绝对不能打进同一个离线包。
- 下载行为必须由目标环境驱动。

### 7.2 环境自检

进入下载流程后，会执行环境自检，检查：

- 当前操作系统和架构
- 包管理器类型
- `tar`
- `curl`
- `dnf/yum` 或 `apt-get`
- `createrepo` / `dpkg-dev`

如果缺少依赖，菜单会显示明确的处理选项，例如：

1. 自动安装缺失依赖
2. 继续执行
0. 返回

### 7.3 工具选择

工具列表来源于 `conf/tools.conf` 及 OS 适配规则。

工具选择界面需要满足：

- 按分类展示
- 支持分页
- 已选择工具可见
- 已存在于离线包中的工具可提示跳过
- 不支持当前目标系统的工具应明确标识

### 7.4 仓库预检

在真正下载前，程序会：

- 检查仓库源可达性
- 过滤不可达源
- 记录实际下载源
- 对工具或包组做预校验

说明：

- 仓库探测应在下载前完成。
- 已判定不可用的仓库源，不应进入真实下载任务。

### 7.5 下载执行

下载阶段按目标系统解析工具与依赖闭包：

- RPM：使用隔离 installroot 进行依赖解析和下载。
- DEB：使用隔离状态避免把“下载主机已安装依赖”误当成已满足。

下载结果要求：

- 成功与失败以“工具”为单位汇总，不以单个依赖包噪声为准。
- 工具失败时必须给出失败原因。
- 常见失败原因需要明确分类，例如：
  - `SOURCE_UNREACHABLE`
  - `PACKAGE_OR_GROUP_NOT_FOUND`
  - `DEPENDENCY_RESOLVE_FAILED`
  - `NO_PACKAGE_ARTIFACT`

### 7.6 打包

下载成功后程序会：

1. 构建本地仓库索引。
2. 生成 `manifest.json`。
3. 生成 `.sha256` 校验文件。
4. 输出离线包到 `output/`。

离线包展示时只应显示“可安装工具列表”，而不是所有依赖包清单。

## 8. 安装模式使用流程

### 8.1 自动识别当前环境

安装模式会尝试识别：

- 当前操作系统
- 当前架构
- 当前内核
- 当前包管理器类型

### 8.2 离线包兼容性检查

当前版本以离线包中的 `manifest.json` 作为可信判断来源，进行：

- 目标 OS 检查
- 架构检查
- 包类型检查

兼容性策略：

- 完全匹配：允许安装
- 同家族兼容：警告并确认
- 不兼容：阻止安装

### 8.3 全量安装

选择“安装全部工具”后：

- 先验证校验文件
- 再解压离线包
- 构建本地仓库索引
- 仅使用本地仓库执行安装

安装过程中，如果本地已安装：

- 可识别已安装工具
- 可识别可升级包
- 可提示跳过或仅升级

### 8.4 选择性安装

选择“选择性安装”后：

- 界面展示的是工具名称，而不是依赖爆炸列表
- 可以只选择离线包中支持的部分工具
- 安装完成后可以继续安装其他工具包，或返回主菜单

### 8.5 本地仓库原则

离线安装必须遵循：

- 不以 `rpm -Uvh *.rpm` 或 `dpkg -i *.deb` 作为默认主路径
- 必须优先通过本地仓库索引执行安装
- 不能在离线环境隐式联网

## 9. 输出文件说明

### 9.1 离线包

典型文件名：

`output/offline_<target_os>_<target_arch>_merged.tar.xz`

例如：

`offline_openEuler22.03_x86_64_merged.tar.xz`

### 9.2 校验文件

每个离线包对应：

`*.sha256`

### 9.3 manifest.json

离线包根目录中包含：

`manifest.json`

主要内容包括：

- 目标系统
- 目标架构
- 包类型
- release 版本
- 工具列表
- 内核依赖列表
- 下载源
- 依赖策略

### 9.4 元数据文件

输出目录下还会有 `.metadata/` 记录，用于展示和增量判断。

## 10. 日志说明

### 10.1 日志位置

主运行日志位于：

`logs/logs_YYYYMMDD.log`

自主验证结果位于：

`logs/autonomous_validation/`

### 10.2 日志内容

当前日志框架应覆盖：

- 目标环境
- 选择的工具
- 仓库源
- 下载动作
- 打包动作
- 安装动作
- 成功/失败统计
- 失败分类
- 最终产物路径

### 10.3 建议排查顺序

出现问题时，建议依次查看：

1. 目标 OS/架构是否选错
2. 工具是否受 `tool_os_rules.conf` 限制
3. 仓库源是否可达
4. 是否存在 `PACKAGE_OR_GROUP_NOT_FOUND`
5. `packages/` 目录中是否有实际产物
6. 离线包中是否存在 `manifest.json`
7. 离线安装时是否被兼容性检查拦截

## 11. 常见问题与处理

### 11.1 `/bin/bash^M: bad interpreter`

原因：

- 脚本被写成了 CRLF 换行。

处理：

- 项目内 shell 脚本必须保持 LF。
- 仓库已通过 `.gitattributes` 固定 `*.sh text eol=lf`。

### 11.2 菜单没有显示，只提示输入

原因：

- 菜单渲染与输入提示顺序不正确。

处理：

- 先检查 `logs/` 中是否有 `dep_check/menu_render`。
- 再检查实际终端显示是否完整。

### 11.3 包下载失败但最后仍然显示成功

原因：

- 旧逻辑按“部分依赖成功”误判为工具成功。

处理：

- 现在应以“工具级成功”作为最终打包依据。
- 失败工具不能计入最终支持工具列表。

### 11.4 `htop` 或某些包组显示失败

可能原因：

- 仓库中无该包/组
- 包组名与当前 OS 实际定义不一致
- 当前工具配置与目标仓库不匹配

处理：

- 先看失败分类是否为 `PACKAGE_OR_GROUP_NOT_FOUND`
- 再核对 `tools.conf`、`tool_os_rules.conf`

### 11.5 安装时提示 `packages` 目录为空

原因：

- 离线包内部未包含真正产物
- 打包前无成功下载结果

处理：

- 不应把这种情况视为安装成功
- 需要回看下载阶段日志与 `manifest.json`

## 13. 配置维护建议

### 13.1 新增工具时

至少检查：

- `conf/tools.conf`
- `conf/tool_os_rules.conf`
- 目标仓库中是否真实存在对应包或包组
- 是否需要特殊依赖策略

### 13.2 修改仓库源时

至少检查：

- `conf/os_sources.conf`
- 网络可达性
- 不同目标 OS 是否误用了同一组仓库

### 13.3 版本发布前

建议至少确认：

- 菜单无乱码
- 中英双语都能工作
- 下载、打包、安装日志可追踪
- `manifest.json` 正确入包
- 兼容性检查可拦截错误安装
- 离线安装不依赖联网

## 14. 路线图说明

当前项目已经支持在 Windows 主机中开发和调试，但“Windows 可分发版本”仍属于后续里程碑，不等同于当前 Bash 脚本在 Windows 上可编辑可运行。

当前优先级仍然是：

1. 完整的离线仓库下载与安装闭环
2. manifest 契约和兼容性检查
3. 自主回归能力
